深入理解RunLoop

Nov 28, 2018 | iOS

RunLoop 是 iOS 和 OSX 开发中非常基础的一个概念，这篇文章将从 CFRunLoop 的源码入手，介绍 RunLoop 的概念以及底层实现原理。同时介绍一下在 iOS 中，苹果是如何利用 RunLoop 实现自动释放池、延迟回调、触摸事件、屏幕刷新等功能的。

一、RunLoop 的概念

RunLoop是与线程相相关的基础组成部分，是一个事件处理循环，用于调度工作和协调接收传入事件。RunLoop的目的是在有工作要做时让线程保持忙碌，在没有工作要做时让线程休眠。RunLoop是iOS事件响应与任务处理最核心的机制，它贯穿iOS整个系统。

OSX/iOS 系统中，提供了两个这样的对象：NSRunLoop 和 CFRunLoopRef。CFRunLoopRef 是在 CoreFoundation 框架内的，它提供了纯 C 函数的 API。NSRunLoop 是基于 CFRunLoopRef 的封装，提供了面向对象的 API。

二、RunLoop 与线程之间的关系

iOS 开发中的线程对象: NSThread。你可以通过 [NSThread mainThread] 来获取主线程；也可以通过 [NSThread currentThread] 来获取当前线程。苹果不允许直接创建 RunLoop，它只提供了两个自动获取的函数：CFRunLoopGetMain() /[NSRunLoop mainRunLoop] 和 CFRunLoopGetCurrent()/[NSRunLoop currentRunLoop]。

线程和 RunLoop 之间是一一对应的，其关系是保存在一个全局的 Dictionary 里。线程刚创建时并没有 RunLoop，如果你不主动获取，那它一直都不会有。RunLoop 的创建是发生在第一次获取时，RunLoop 的销毁是发生在线程结束时。你只能在一个线程的内部获取其 RunLoop，当然，主线程除外。

上图显示了RunLoop和各种源的概念结构，Run Loop的处理两大类事件源：Timer Source和Input Source(包括performSelector *方法簇、Port或者自定义的Input Source）。Input Source传递异步事件，通常是来自另一个线程或其他应用程序的消息。Timer Source传递同步事件，这些事件在计划的时间或重复的间隔发生。两种类型的源都使用特定于应用程序的处理程序例程来处理事件到达时的事件。

Input Source将异步事件传递给相应的处理程序，程序将会一直运行直到最终触发runUntilDate:方法，然后在相关RunLoop中调用Exit。Timer Source将事件传递到其处理程序例程，但不会导致RunLoop退出。

这里有个概念叫CommonModes：一个 Mode 可以将自己标记为”Common”属性（通过将其 ModeName 添加到 RunLoop 的 “commonModes” 中）。每当 RunLoop 的内容发生变化时，RunLoop 都会自动将 _commonModeItems 里的 Source/Observer/Timer 同步到具有 “Common” 标记的所有Mode里。

应用场景举例：主线程的 RunLoop 里有两个预置的 Mode：kCFRunLoopDefaultMode 和 UITrackingRunLoopMode。这两个 Mode 都已经被标记为”Common”属性。DefaultMode 是 App 平时所处的状态，TrackingRunLoopMode 是追踪 ScrollView 滑动时的状态。当你创建一个 Timer 并加到 DefaultMode 时，Timer 会得到重复回调，但此时滑动一个TableView时，RunLoop 会将 mode 切换为 TrackingRunLoopMode，这时 Timer 就不会被回调，并且也不会影响到滑动操作。

有时你需要一个 Timer，在两个 Mode 中都能得到回调，一种办法就是将这个 Timer 分别加入这两个 Mode。还有一种方式，就是将 Timer 加入到顶层的 RunLoop 的 “commonModeItems” 中。”commonModeItems” 被 RunLoop 自动更新到所有具有”Common”属性的 Mode 里去。
